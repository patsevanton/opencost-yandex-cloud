# Стратегия получения и проверки данных для custom-pricing-configmap.yaml

Данный документ описывает, **откуда брать** значения для ConfigMap кастомных цен OpenCost (Yandex Cloud, ₽) и **как проверять** их актуальность и корректность.

## 1. Назначение полей ConfigMap

| Ключ | Описание | Единица в ConfigMap | Как использует OpenCost |
|------|----------|---------------------|-------------------------|
| `provider` | Идентификатор провайдера | строка `custom` | Выбор модели расчёта |
| `description` | Подпись | строка | Информационно |
| `CPU` | Стоимость vCPU | ₽/мес за 1 vCPU | Делит на 730 → ₽/час за 1 vCPU |
| `RAM` | Стоимость памяти | ₽/мес за 1 ГБ | Делит на 730 → ₽/час за 1 ГБ |
| `storage` | Стоимость диска | ₽/мес за 1 ГБ | Делит на 730 → ₽/час за 1 ГБ |
| `zoneNetworkEgress` | Исходящий трафик в зоне | ₽ за 1 ГБ | Без деления на 730 |
| `regionNetworkEgress` | Исходящий трафик в регионе | ₽ за 1 ГБ | — |
| `internetNetworkEgress` | Исходящий трафик в интернет | ₽ за 1 ГБ | — |
| `LBIngressDataCost` | Load Balancer, входящий трафик | ₽ за единицу | — |
| `FirstFiveForwardingRulesCost` | Первые 5 правил LB | ₽ | — |
| `AdditionalForwardingRuleCost` | Доп. правила LB | ₽ | — |

**Важно:** для CPU, RAM и storage в ConfigMap указываются **месячные** ставки (₽/мес). Почасовая ставка = значение / 730. Источник тарифов должен давать либо почасовые цены (тогда умножаем на 730), либо месячные (подставляем как есть).


## 2. Получение данных (источники)

### 2.1. Официальная документация Yandex Cloud (основной источник)

Тарифы публикуются в документации. Актуальные страницы:

- **Compute Cloud (ВМ):** [Тарификация Compute Cloud](https://cloud.yandex.ru/docs/compute/pricing) — vCPU, RAM, диск, исходящий трафик.
- **Managed Service for Kubernetes:** [Тарификация Managed Service for Kubernetes](https://cloud.yandex.ru/docs/managed-kubernetes/pricing) — если ноды управляются сервисом (мастер и т.д.).
- **Сеть:** трафик обычно описан на странице Compute или в [Сеть — тарификация](https://cloud.yandex.ru/docs/vpc/pricing).

**Действия:**

1. Открыть страницу тарификации для вашего региона (например, ru-central1-a).
2. Выписать почасовые цены:
   - vCPU, ₽/час за 1 vCPU
   - RAM, ₽/час за 1 ГБ
   - Диск (стандартное/быстрое сетевое и т.д.), ₽/час за 1 ГБ
3. Исходящий трафик: ₽ за 1 ГБ (часто без деления на часы).
4. Пересчитать в формат ConfigMap:
   - `CPU = vCPU_₽/час * 730`
   - `RAM = RAM_₽/час * 730`
   - `storage = disk_₽/час * 730`
   - `zoneNetworkEgress`, `regionNetworkEgress`, `internetNetworkEgress` — как в документации (за 1 ГБ).

Регион и тип диска должны соответствовать вашему кластеру (например, стандартные диски нод, тот же регион).

### 2.2. Billing API — каталог SKU (тарифы list price)

В Billing API есть **SkuService** ([Billing API — Sku](https://yandex.cloud/ru/docs/billing/api-ref/Sku/)): методы `Get`, `List` возвращают каталог SKU и тарифы (list price). Это подходит для программного получения **справочных** цен.

**Ограничения:**

- Нужен IAM-токен и права на биллинг.
- Структура ответа (привязка SKU к vCPU/RAM/диску/трафику) может требовать маппинга по `sku_id` или названию сервиса/SKU.
- Цены в ответе могут быть в копейках или в рублях — уточнять по [документации API](https://yandex.cloud/ru/docs/billing/api-ref/Sku/list).

**Идея использования:**

1. Вызвать `Sku.List` (с фильтром по сервису Compute Cloud при наличии).
2. Найти SKU для: vCPU, RAM, диска, исходящего трафика.
3. Преобразовать единицы (час/месяц, за единицу ресурса) в формат ConfigMap.
4. Результат можно использовать для автоматического обновления ConfigMap (скрипт + CI или ручной запуск).

В репозитории можно завести скрипт `scripts/fetch_yandex_sku_prices.py`, который по IAM-токену вызывает Billing API, находит нужные SKU и выводит готовые значения для `CPU`, `RAM`, `storage`, egress (или весь фрагмент `data` для ConfigMap).

### 2.3. Ручная таблица (CSV/таблица в репозитории)

Если API не используется, достаточно один раз зафиксировать источник и формулы в репозитории:

- Файл `docs/pricing-sources.md` (или раздел в этом документе) с ссылками на страницы тарификации и датой проверки.
- В комментариях в `custom-pricing-configmap.yaml` указать расчёт, например: `# 1.2852 ₽/vCPU-час * 730`.

Так проще при следующем обновлении повторить расчёт и проверить изменения.


## 3. Проверка данных

### 3.1. Проверка формата и формул (локально)

Перед применением ConfigMap стоит вручную проверить:

- Наличие обязательных ключей: `provider`, `CPU`, `RAM`, `storage` (опционально egress и LB).
- Что значения — числа в разумных диапазонах (CPU/RAM/storage положительные, не нули).
- Согласованность «почасовая ставка × 730 = месячная» для CPU/RAM/storage, если в комментариях указаны почасовые эталоны.

ConfigMap можно взять из файла `custom-pricing-configmap.yaml` или из кластера: `kubectl get configmap custom-pricing-model -n opencost -o yaml`.

### 3.2. Проверка внутри кластера (OpenCost использует ConfigMap)

1. Убедиться, что ConfigMap применён и под OpenCost перезапущен (или дождаться перечитывания конфига, если поддерживается):
   ```bash
   kubectl get configmap custom-pricing-model -n opencost -o yaml
   kubectl rollout restart deployment/open cost -n opencost   # имя по факту
   ```
2. Проверить метрики OpenCost:
   - `node_cpu_hourly_cost`, `node_ram_hourly_cost`, `node_total_hourly_cost` — должны быть в разумных величинах (в USD по умолчанию; при custom pricing единицы зависят от того, как OpenCost выводит их в метриках).
   - Либо открыть UI OpenCost и убедиться, что стоимость нод и аллокаций не нулевая и не аномально большая.

### 3.3. Сверка с фактическими расходами (биллинг Yandex Cloud)

Чтобы проверить, что расчёт OpenCost по тарифам близок к факту:

1. **Получить фактические списания** за период (например, месяц):
   - Экспорт детализации в CSV в Object Storage (см. [yandex-cloud-billing-api-charges.md](yandex-cloud-billing-api-charges.md)).
   - Либо отчёт через Yandex Query по выгруженным данным.
2. **Получить расчёт OpenCost** за тот же период:
   - Allocation API за месяц (или сумма по дням), агрегат по кластеру/нодам.
   - Сравнить с суммой по биллингу по сервисам Compute Cloud / Managed Kubernetes (ноды, диски, трафик).

Расхождение возможно из-за:

- разницы list price (тарифы в ConfigMap) и фактических скидок/кредитов в биллинге;
- неучтённых статей (мастер Managed Kubernetes, резервные копии, объектное хранилище и т.д.);
- округления и разницы момента учёта (час/день).

Имеет смысл зафиксировать в репозитории шаги сверки (какой запрос Allocation, какой отчёт биллинга) и периодически повторять проверку при изменении тарифов или конфигурации кластера.


## 4. Чек-лист обновления тарифов

| Шаг | Действие |
|-----|-----------|
| 1 | Открыть актуальную страницу тарификации Yandex Cloud (Compute/Kubernetes, нужный регион). |
| 2 | Выписать почасовые цены vCPU, RAM, диск (и при необходимости трафик). |
| 3 | Посчитать месячные: значение × 730 для CPU, RAM, storage. |
| 4 | Обновить `custom-pricing-configmap.yaml`, сохранить комментарии с формулой (₽/час и × 730). |
| 5 | Применить ConfigMap, при необходимости перезапустить OpenCost. |
| 6 | Проверить метрики/UI OpenCost и при возможности сверить с биллингом за период. |
